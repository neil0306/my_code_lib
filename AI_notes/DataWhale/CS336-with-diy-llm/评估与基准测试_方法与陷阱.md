# 评估与基准测试：方法与陷阱

---

## 1. 评估不是一个分数

同一个模型在不同评估设置下会得到不同结论：

- 任务覆盖不同；
- 提示策略不同；
- 打分函数不同；
- 成本约束不同。

核心观点：
**先定义“要回答的问题”，再设计评估。**

---

## 2. 评估目标四分法

- 用户视角：哪个模型更好用（体验、延迟、价格）？
- 研究视角：模型能力是否真实提升？
- 安全视角：风险类型与触发边界是什么？
- 工程视角：哪条训练改动真正有效？

---

## 3. 评估设计四要素

### 3.1 输入（Input）

- 是否覆盖真实任务分布与长尾输入；
- 是否包含对抗样本和边界样本；
- 多轮任务是否考虑上下文依赖。

### 3.2 调用方式（Protocol）

- zero-shot / few-shot / CoT 是否固定；
- 是否允许工具调用；
- 是评模型本体，还是评“模型 + agent 框架”。

### 3.3 输出打分（Scoring）

- 客观题：准确率、pass@k；
- 开放题：人工评估、LLM-as-a-judge、代理指标；
- 需把成本、延迟与准确率联合考虑。

### 3.4 结果解释（Interpretation）

- 分数是否统计显著；
- 是否存在数据泄露/刷榜；
- 是否达到可部署门槛，而不只是“实验室更高分”。

---

## 4. 困惑度（Perplexity）该怎么用

定义：

$$
\text{Perplexity}=\left(\frac{1}{p(D)}\right)^{1/N}
$$

直觉：

- 越低表示模型越“确信”测试文本；
- 适合看语言建模基础能力与缩放趋势。

适用场景：

- 训练过程监控；
- 模型容量/数据规模实验；
- 同分布语料上的对比分析。

局限：

- 与任务可用性并非一一对应；
- 黑盒 API 往往拿不到可靠 token 概率；
- 容易被分词器差异影响。

---

## 5. 成本-性能联合评估

只看准确率会误导。至少要看：

- 质量（准确率、通过率、胜率）；
- 成本（每百万 token 价格、总推理成本）；
- 时延（TTFT、token latency、p95/p99）。

可用 Pareto 视角：

- 找“同等成本下更高质量”或“同等质量下更低成本”的前沿模型。

---

## 6. 常见误用 -> 正确写法

- 误用：只用单一 benchmark 排名做结论。  
  正确：任务簇评估 + 误差条 + 多维指标联合解读。
- 误用：评测集反复调参到过拟合。  
  正确：保留隐藏集，限制评测回看次数。
- 误用：只报告均值。  
  正确：同时报告方差、置信区间、失败样本分布。
- 误用：忽略成本和时延。  
  正确：质量/成本/时延三维一起给决策。

---

## 7. 实战评估模板

1. 定义目标场景与风险约束；
2. 设计任务集（主任务 + 长尾 + 对抗）；
3. 固定调用协议（prompt、工具、采样参数）；
4. 运行评测并记录全量日志；
5. 做分层分析（任务类型、长度、语言、难度）；
6. 输出决策报告（是否上线、上线保护策略、后续实验）。


## 8. 代码化模板：评估实验配置（可直接改）

```yaml
# eval_config.yaml
experiment: "eval_v1"
models:
  - name: "model_a"
    endpoint: "..."
  - name: "model_b"
    endpoint: "..."

tasks:
  - name: "gsm8k"
    metric: "accuracy"
  - name: "humaneval"
    metric: "pass@1"
  - name: "safety_refusal"
    metric: "refusal_precision"

decoding:
  temperature: 0.0
  top_p: 1.0
  max_tokens: 1024

cost_metrics:
  record_ttft: true
  record_token_latency_p95: true
  record_usd_per_1m_tokens: true

significance:
  bootstrap_samples: 2000
  ci_level: 0.95
```

**常见误用 -> 正确写法**
- 误用：每次实验改 prompt 但不记录版本。  
  正确：`prompt_template_version` 必须进配置文件。

---

## 10. 一句话总结

好的评估不是“刷高分”，而是：  
**在真实约束下，可靠地区分“可部署进步”与“指标幻觉”。**
