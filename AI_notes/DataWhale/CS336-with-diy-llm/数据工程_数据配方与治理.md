# 数据工程：数据配方与治理

---

## 1. 数据工程在 LLM 中的定位

数据工程不是“收集数据”这么简单，而是一个端到端系统：

- 数据获取（crawl / 许可 / 来源追踪）；
- 数据清洗（去重、去噪、格式归一）；
- 数据配比（通用、代码、数学、长文本、多语种）；
- 数据安全与合规（版权、隐私、偏见、有害内容）；
- 阶段化投喂（预训练 / 中期训练 / 后训练）。

核心直觉：
**模型参数在学什么，主要由数据分布决定。**

---

## 2. 三阶段数据策略

### 2.1 预训练数据

目标：学习通用语言规律与世界知识。  
特征：规模大、来源广、噪声高。

常见来源：

- 网页文本（如 Common Crawl）；
- 书籍、百科；
- 代码仓库；
- 学术与技术文档。

### 2.2 中期训练数据

目标：定向增强（数学、代码、长上下文、专业能力）。  
特征：规模小于预训练，但质量要求更高。

### 2.3 后训练数据

目标：行为对齐（指令遵循、安全性、偏好一致性）。  
特征：高质量示范、偏好对、可验证任务数据（如 RLVR 数学/代码）。

---

## 3. 数据源选择的工程标准

### 3.1 代码数据

价值：

- 提升程序生成与调试能力；
- 增强多步推理与结构化表达能力。

工程筛选：

- 去重（仓库级 + 文件级 + 片段级）；
- License 过滤（仅保留可用许可证）；
- 质量规则（过滤自动生成、低信息密度、二进制伪文本）。

### 3.2 书籍数据

价值：

- 提供长程依赖与篇章结构；
- 强化叙事和论证连贯性。

工程筛选：

- 版权合规优先；
- 章节边界保留，避免粗暴切分破坏上下文；
- 与网页短文本混合时注意长度分布平衡。

### 3.3 数学与科学数据

价值：

- 强化符号推理、公式表达、严谨问答。

工程筛选：

- 公式解析与文本清洗（LaTeX、MathML）；
- 问答对齐与答案可验证性检查；
- 防止“只学格式不学内容”的模板污染。

---

## 4. 合成数据：为什么有效，何时危险

合成数据常用于：

- 稀缺领域补样；
- 指令风格统一；
- 低成本扩充难样本。

它有效的原因：

- 教师模型可生成结构清晰的示范轨迹；
- 能快速覆盖长尾任务模板。

风险：

- 教师偏见被复制放大；
- 风格单一导致学生模型“口吻塌缩”；
- 事实性错误被系统性蒸馏。

工程原则：

- 合成数据必须与真实数据混合；
- 做质量门控（规则 + 模型打分 + 人审抽样）；
- 追踪数据血缘（source id / 生成器版本 / 过滤规则版本）。

---

## 5. 数据质量维度与可观测指标

关键维度：

- 正确性（correctness）；
- 一致性（consistency）；
- 完整性（completeness）；
- 多样性（diversity）；
- 去重率（dedup ratio）；
- 覆盖度（coverage）。

建议仪表盘：

- 按来源统计 token 占比；
- 按领域统计质量分数分布；
- 按阶段统计 loss 改善贡献；
- 按安全规则统计拦截率。

---

## 6. 常见误用 -> 正确写法

- 误用：只追求 token 总量。  
  正确：同时优化质量、覆盖与配比。
- 误用：把去重当一次性离线任务。  
  正确：建立持续去重与增量检测流水线。
- 误用：合成数据大规模替代真实数据。  
  正确：真实数据奠基，合成数据做定向增强。
- 误用：忽略版权与隐私，先训再补救。  
  正确：把合规检查前置到 ingestion 阶段。

---

## 7. 可执行数据流水线（最小版）

1. Source Ingestion：抓取/导入 + 元数据登记；
2. Compliance Gate：版权、隐私、许可证过滤；
3. Cleaning：去重、去噪、格式统一；
4. Scoring：质量/领域分类打分；
5. Mixing：按阶段目标做数据配比；
6. Training Feedback：用 loss 与下游指标反哺配方；
7. Iteration：版本化数据集，滚动迭代。


## 8. 代码化模板：数据配方表（可直接改）

```yaml
# data_recipe.yaml
version: "v1"
stage: "pretrain"
mixture:
  web:
    target_ratio: 0.62
    quality_threshold: 0.70
  code:
    target_ratio: 0.15
    license_allowlist: ["MIT", "Apache-2.0", "BSD-3-Clause"]
  books:
    target_ratio: 0.10
    public_domain_only: true
  math_science:
    target_ratio: 0.13
    require_formula_parse: true

filters:
  dedup:
    enabled: true
    minhash_jaccard: 0.85
  pii:
    enabled: true
    block_types: ["email", "phone", "id_card"]
  safety:
    enabled: true
    toxic_threshold: 0.80

quality_gates:
  min_avg_quality_score: 0.75
  max_near_dup_ratio: 0.12
  min_domain_coverage: 0.80
```

**常见误用 -> 正确写法**
- 误用：配方只写比例，不写质量门槛。  
  正确：比例 + 质量阈值 + 过滤规则一起版本化。

## 9. 代码化模板：数据流水线检查脚本（伪代码）

```python
from dataclasses import dataclass
from typing import Dict

@dataclass
class BatchStats:
    tokens: int
    avg_quality: float
    near_dup_ratio: float
    pii_hit_ratio: float
    domain_coverage: Dict[str, float]


def validate_batch(stats: BatchStats) -> list[str]:
    errors = []
    if stats.avg_quality < 0.75:
        errors.append("quality score below threshold")
    if stats.near_dup_ratio > 0.12:
        errors.append("near duplicate ratio too high")
    if stats.pii_hit_ratio > 0.0:
        errors.append("PII leakage risk")
    if min(stats.domain_coverage.values()) < 0.80:
        errors.append("domain coverage too low")
    return errors


def should_publish_dataset(stats: BatchStats) -> bool:
    return len(validate_batch(stats)) == 0
```

**工程注意**
- 检查脚本要跑在“切分后样本”上（train/val/test），否则上线后会出现分布反转。
- 每次配方调整都要回填变更说明（为什么调、预期影响、回滚条件）。

---

## 10. 一句话总结

数据工程的本质不是“多收集”，而是：  
**用可追踪、可迭代、可验证的数据配方，把算力转化成稳定能力。**
